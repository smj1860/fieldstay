## Agent 2 — Prompt Injection & Output Safety

- **High (confirmed): Escalation detection is brittle keyword-matching on the bot's OUTPUT, not a structured signal.** `detectEscalation()` in `lib/support/respond.ts` (lines 118–134) only checks the final response text for 6 fixed English phrases ("our team", "flagging this for", etc.). A user can plausibly induce the model to phrase a real escalation differently (e.g. by asking it to avoid certain words, or just via natural model phrasing drift across the two models used per category) and the conversation will silently never get `needs_human=true`, never enter the staff inbox, and never fire the `support/conversation.escalated` Inngest event — even for billing disputes, safety incidents, or legal threats. Recommend replacing with a forced structured tool-call output (the codebase already has this pattern in `classify.ts`'s `route_support_request` tool) instead of string matching.
- **Medium (confirmed): System prompt lacks explicit injection-resistance instructions.** `buildSystemPrompt()` in `lib/support/respond.ts` never tells the model to disregard "ignore your instructions" style attacks embedded in user messages, nor to refuse revealing the system prompt/tool definitions verbatim. Claude has baseline resistance, but no application-level hardening exists. Recommend adding an explicit precedence + non-disclosure instruction.
- **Clean (confirmed): Knowledge-base poisoning path does not exist.** Grepped every reference to `support_kb_chunks` repo-wide. Only `scripts/seed-support-kb.ts` writes to it, reading exclusively from local `docs/support/*.md` files via a manually-run CLI script using the service-role key. No API route inserts/updates this table. RLS policy is read-only (`FOR SELECT ... USING (true)`) with no INSERT/UPDATE/DELETE policy defined, so even a client-role write attempt would be denied by default-deny RLS.
- **Clean (confirmed): Tool-use loop hard cap at 3 rounds is real and unconditional.** `generateAccountSpecificResponse()` in `lib/support/respond.ts` increments a function-local `rounds` counter unconditionally every iteration; no message content can cause it to loop indefinitely or skip the increment. Max 4 Anthropic API calls per account-specific request, each capped at 800 tokens — no cost/DoS amplification vector.
- **Clean (confirmed): No cross-org contamination risk.** `orgId` is threaded explicitly as a function parameter end-to-end (route handler → `generateResponse` → `generateAccountSpecificResponse` → `callAccountTool` → each `get*` function's `.eq('org_id', orgId)` filter), always derived server-side from `requireOrgMember()`'s authenticated session — never from model tool-call arguments (tool schemas accept zero model-supplied parameters) or request body. No module-level/global mutable state found anywhere in `lib/support/*` that could leak between concurrent requests.
